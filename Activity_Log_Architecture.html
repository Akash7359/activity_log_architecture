07 Sync Mechanism — React Background Worker

Sync Trigger Conditions:
  • Every 30 seconds
  • OR unsynced logs ≥ 100
  • OR user comes online
  • OR before tab close
  • OR on visibility change

Batch Processing:

  1. Fetch logs WHERE synced = false ORDER BY timestamp ASC LIMIT 100
  2. Create batch payload:
       {
         batch_id:   UUID,
         total_logs: count,
         logs:       [ ...Full JSON logs ]
       }

  3. Generate SHA256 integrity hash:
       audit_id + timestamp + user.id + trace.trace_id

  4. POST /api/v1/activity/logs/batch
       Headers: Authorization · X-Batch-ID · X-Integrity-Hash

On Success (200 OK):
  • Mark synced = true
  • Store server acknowledgement timestamp
  • Store server trace_id

On Failure — Exponential Backoff:
  retry 1  →  30 sec
  retry 2  →   2 min
  retry 3  →   5 min
  retry 4  →  10 min
  retry 5  →  Stop & flag

Security Controls:
  • HTTPS only
  • Mask device.ip_address (optional)
  • No raw PII in storage
  • Validate log_integrity_hash before send

08 Backend Ingestion Architecture — Laravel

Endpoint:  POST /api/v1/activity/logs/batch

Pipeline:
  1. Validate JSON schema
  2. Verify integrity hash (SHA256)
  3. Rate limit check (per client / IP)
  4. Add server_timestamp
  5. Attach server trace_id
  6. Push logs → Queue (Redis / Database Queue)
  7. Return 200 OK immediately (non-blocking)

Queue Worker (async):
  • Consume batch from queue
  • Persist structured columns → MySQL (hot)
  • Serialize full JSON → S3 (cold archive)
  • Fan-out risk feed → Kafka → FRM Consumer

11 SIEM & Fraud Detection Layer

Fraud Rule Examples:

  1. Rapid Click
     click_speed_ms < 100
     → risk_score +10

  2. Multiple IP Change
     3+ distinct IPs in 10 mins
     → risk_score +25

  3. API Failures
     status_code ≥ 400  repeated 5× in session
     → anomaly_detected = true

  4. Bot Behavior
     mouse_movement_pattern = 'BOT_LIKE'
     → risk_score +30

Escalation:
  IF risk_score > 70:
    → anomaly_detected  = true
    → Create Fraud Case
    → Dispatch Alert

SIEM Alert Escalation Model

LOW< 30Log only—
MEDIUM30–59Dashboard monitoringAdmin Dashboard
HIGH60–79Alert security teamEmail
CRITICAL≥ 80Create Incident + NotifyEmail + SMS + Webhook

  Option 1:  Risk Engine → Push Alerts → Enterprise SIEM (Syslog / Webhook)
  Option 2:  Elasticsearch Cluster acts as internal SIEM index
  Option 3:  Wazuh / ELK consumes logs directly from Message Broker

12 FRM Severity Rule Engine Configuration

All rules are centrally configurable and can be enabled/disabled without redeployment.

Rule IDEvent TypeCondition ImpactLikelihoodRisk Δ SeverityAlert ChannelAuto CaseActive
R001UI_ACTIVITYrapid_click_flag = true LOWMEDIUM+10 LOWNONENo✓
R002API_FAILUREstatus_code ≥ 400 AND count ≥ 5 MEDIUMMEDIUM+20 MEDIUMDASHBOARDNo✓
R003MULTI_IPdistinct_ip ≥ 3 within 10 min HIGHHIGH+30 HIGHEMAILYes✓
R004BOT_PATTERNmouse_pattern = BOT HIGHHIGH+40 CRITICALEMAIL + SMSYes✓

Rule Execution Logic

For each incoming log:

  1. Evaluate all ACTIVE rules
  2. If condition_expression = TRUE:
       → Accumulate risk_score_increment
       → Apply immediate severity (if threshold breach)
       → Trigger alert_channel if action_required = true
       → Create case if auto_case = true

  3. After all rules evaluated:
       Determine FINAL severity from thresholds:

       total_risk_score < 30   →  LOW
       total_risk_score 30–59  →  MEDIUM
       total_risk_score 60–79  →  HIGH
       total_risk_score ≥ 80   →  CRITICAL

14 Compliance & Security Controls

  • No raw PII stored — sensitive fields masked before persistence
  • SHA256 integrity hash per log (verified on ingestion)
  • Immutable archive in S3 (Object Lock / WORM)
  • Audit trail retention: 7 years (regulatory compliance)
  • HTTPS only — no plaintext transmission
  • Role-based access control (RBAC) on all storage layers
  • Service-to-service authentication via mTLS
  • Daily snapshot + weekly offline backup
  • OpenTelemetry trace correlation for full request lineage

       └── MM/

Core Microservices

  • Log Ingestion Service — receives batched logs from all application clients
  • Queue Broker — asynchronous, fault-tolerant processing
  • Risk Engine — applies severity matrix (Impact × Likelihood)
  • Rule Engine — evaluates configurable fraud rules against each event
  • Alert Service — dispatches Email / SMS / Webhook notifications
  • Case Management — opens and tracks fraud investigation tickets
  • Archive Service — compresses and writes immutable logs to NAS / MinIO

Security & Compliance Controls

  • Immutable write — append-only storage (WORM)
  • SHA256 integrity verification per log
  • RBAC — role-based access control
  • mTLS — service-to-service authentication
  • RAID storage redundancy
  • Daily snapshot + weekly offline backup
